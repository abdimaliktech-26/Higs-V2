- Public Client Portal — Stage 4 Step 4: Request Notifications (built on Steps 1-3's request/upload/review foundation; `PortalNotification` only — the staff `Notification` model is never touched)
  - Notification trigger map: new request → all active grants (`document_request`, links to `/portal/upload?client=&request=`); upload received → uploader only (`upload_received`, same upload link); upload approved → all active grants (`upload_approved`, links to `/portal/documents?client=`); needs replacement → all active grants (`needs_replacement`, upload link); due tomorrow / due in 3 days → all active grants (`due_tomorrow`/`due_in_3_days`, upload link). `APPROVED` is treated as the terminal completion notification for this step — packet-wide completion is deferred to the future Upload Checklist step.
  - Recipient scoping: `notifyActivePortalUsersForClient`/`notifySinglePortalUser` (`src/lib/portal/notifications.ts`) filter at the query level — `status: "ACTIVE"`, `revokedAt: null`, `OR: [{expiresAt: null}, {expiresAt: {gt: now}}]` — so expired/suspended/revoked grants are never fetched, not merely filtered client-side. Both accept an optional transaction client (mirroring `createPortalAuditEvent(input, tx?)`) so the review action and upload route commit their notification writes atomically with the rest of the transaction.
  - Upload-received confirmation is deliberately scoped to only the uploading `PortalUser` — no fan-out, no staff `Notification` row created for this event.
  - Due-date reminders: a deduplicated **on-demand scan**, not a cron or background worker (none exists anywhere in this app) — triggered synchronously whenever the portal dashboard or notifications page renders (`generatePortalDueDateReminders`). Uses discrete calendar-day windows (`[+1 day, +2 days)` and `[+3 days, +4 days)`), not continuous ranges, so a request 2 days out, far in the future, or already past due never generates these two reminder types.
  - Deduplication: before creating a reminder, a per-recipient check (`portalNotification.findFirst` with `metadata: {path: ["requestId"], equals}`, mirroring the staff `generateNotifications` JSON-path pattern) prevents duplicate active reminders across repeated page loads for the same portal user, client, request, and reminder type.
  - Read-state: `markPortalNotificationRead` loads the notification and compares its `portalUserId` against the authenticated caller (`requirePortalAuth()`), throwing "Notification not found" (no existence leak) on any mismatch — a portal user can never mark another user's notification read.

- Public Client Portal — Stage 4 Step 5: Upload Checklist & Completion Tracking (built on Steps 1-4's request/upload/review/notification foundation; finishes the request→review lifecycle before the future Templates & Forms Manager work)
  - Checklist definition and formula (`computeChecklistSummary`, `src/lib/actions/portal-document-requests.ts`): computed from `PortalDocumentRequest` rows only, via two direct `count()` queries — `requiredTotal` = `isRequired: true` AND `status != CANCELLED`; `requiredCompleted` = `isRequired: true` AND `status: APPROVED`; `remaining = requiredTotal - requiredCompleted`; `completionPercent = requiredTotal === 0 ? 0 : round(requiredCompleted / requiredTotal * 100)` (returns a clean `0`, never `NaN`, when there are no required requests).
  - Required-only counting / cancelled-and-optional exclusions: optional (`isRequired: false`) requests never enter either count; cancelled requests never enter the total — both enforced directly in the `count()` `where` clauses, not filtered after the fact.
  - `PacketDocument` linkage behavior: on `APPROVED`, the linked `PacketDocument` (`status: "completed"`, `completedAt: now`) is updated **only** when `PortalDocumentRequest.packetDocumentId` is explicitly set — never inferred from `packetId` alone. Ownership is re-verified at review time (same organization, same client, and same `packetId` if the request has one) rather than trusting the FK pairing stored at request-creation time. `NEEDS_REPLACEMENT`/`UNDER_REVIEW`/`SUBMITTED`/`PENDING`/`CANCELLED` never touch `PacketDocument`. An already-`completed` `PacketDocument` is left untouched (no re-write, no duplicate audit event) — the approval itself still succeeds.
  - Ad-hoc request behavior: requests with no `packetDocumentId` (e.g. "send a photo of your insurance card") count toward the portal-native checklist like any other required request, but approving them never touches `PacketDocument` or packet completion — verified live (see below) by approving an ad-hoc request and confirming both `packet_documents` rows were byte-for-byte unchanged.
  - Duplicate active request guard: `createPortalDocumentRequest` now wraps its conflict check + create in `prisma.$transaction` and rejects creation if another request with the same `packetDocumentId` already has status `PENDING`/`SUBMITTED`/`UNDER_REVIEW`/`NEEDS_REPLACEMENT`. A prior `APPROVED` or `CANCELLED` request never blocks a new one (excluded by the query itself). No database-level uniqueness constraint was added — the status condition can't be expressed safely as one — so this is application-level validation with the transaction narrowing (not eliminating) the race window, exactly as scoped.
  - Packet completion provenance: new `AuditAction.PACKET_DOCUMENT_COMPLETED_VIA_PORTAL` (migration `20260711025608_add_packet_document_completed_via_portal_audit_action`), written inside the same review transaction — kept deliberately distinct from the existing manual `PACKET_DOCUMENT_STATUS_CHANGED` action, so a provider can always tell a portal-driven completion from a staff-driven one. `createAuditEvent` gained an optional transaction-client parameter (mirroring `createPortalAuditEvent(input, tx?)`) so this write commits atomically with the approval. Metadata is limited to exactly `{requestId, packetDocumentId, supportingDocumentId, transition: "completed"}` — no client name, document title, feedback text, or PHI.
  - Portal checklist UI (`/portal/upload`): a "Required Documents" summary card — X of Y completed, a progress bar, and a remaining-count line in plain client-friendly language — added above the existing per-request cards. No internal validation results, audit status, compliance scoring, or reviewer identity beyond what was already client-visible.
  - Staff checklist/status summary (`/clients/[id]/portal-access`, existing Document Requests card): the same required-completed/total/remaining/percent summary plus a request-status breakdown (Pending/Submitted/Under Review/Needs Replacement/Approved counts) — reusing the existing card, no new analytics dashboard.
  - Verification: `npx prisma validate` clean, `npm run lint` clean, `npm run build` clean, `npm test` **193/193 passed** (167 prior + 26 new — checklist formula/exclusions, packet linkage incl. cross-org/cross-client/cross-packet rejection and already-completed handling, and the duplicate-request guard). Live verification against the real running server + database (real NextAuth credentials login via curl, real portal session, seeded packet with two linked `PacketDocument`s + one ad-hoc request, fixtures cleaned up afterward): staff checklist read 0/3 (0%) before any approval, 1/3 (33%) after the linked request's approval was simulated at the database layer, and 2/3 (67%) after the ad-hoc request was also approved; the portal upload page independently rendered the same "2 required documents remaining" and progress value `33` matching the database; both `packet_documents` rows were confirmed completely unchanged by the ad-hoc approval; a real `PENDING` row was confirmed to exist in exactly the state the duplicate-guard query targets; staff login, staff dashboard, portal login, portal notifications, and the health endpoint all remained 200 throughout.
  - Scalability note: the checklist runs two `count()` queries per page load (portal upload page and staff portal-access page), with no caching — fine at current portal traffic, worth revisiting only if request volumes grow significantly.
