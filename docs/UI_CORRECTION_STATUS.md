- Templates & Forms Manager — Stage 5 Step 4a: Conditional Logic Foundation (no UI, no runtime enforcement yet — fourth step of the phased Stage 5 rebuild, first of a four-part split; built on Step 2's real `DocumentTemplateField` definitions)
  - `TemplateConditionGroup` / `TemplateCondition`: normalized relational models (deliberately **not** a JSON leaf array — dependency protection and operator validation need to be queryable, not opaque blobs). `ConditionPurpose` (`FIELD_VISIBILITY`, `FIELD_REQUIREDNESS`, `DOCUMENT_INCLUSION`, `DOCUMENT_REQUIREDNESS`, `VALIDATION_RULE_APPLICABILITY`), `ConditionLogicOperator` (`AND`/`OR`), `ConditionOperator` (12 operators), `ConditionSourceType` (`TEMPLATE_FIELD`, `CLIENT_IS_MINOR`, `PACKET_PROGRAM_CODE`, `PACKET_TYPE`).

Server evaluation is authoritative; the same evaluator is meant for client-side preview only, once a UI exists.

Unsupported operator/type pairs, malformed dates, non-numeric values on numeric operators, empty arrays, and unknown pseudo-fields are all rejected before a row is ever written.
  - Dependency protection: deleting or renaming a `fieldKey` that any condition references is blocked with an accurate dependency count and the affected purposes (`updateDocumentTemplateField` / `deleteDocumentTemplateField` in `document-template-fields.ts`); a display-name-only change remains unaffected.

- Templates & Forms Manager — Stage 5 Step 4b: Document-Owned Conditional Logic and Cross-Document References (backend foundation only; runtime materialization remains deferred to Step 4c)
  - Cross-document source model: `TemplateCondition.sourcePacketTemplateDocumentId` anchors a `TEMPLATE_FIELD` reference to a specific `PacketTemplateDocument.id`, removing the ambiguity created when sibling mapped documents reuse the same `fieldKey`. The relation is indexed and exposes incoming references through `PacketTemplateDocument.referencedByConditions`.
  - Document-owned condition groups: root `TemplateConditionGroup` rows may now be owned by a `PacketTemplateDocument`, with compatible purposes constrained to exactly `DOCUMENT_INCLUSION` and `DOCUMENT_REQUIREDNESS`. Existing field owners remain constrained to `FIELD_VISIBILITY`/`FIELD_REQUIREDNESS`, and incompatible or multi-owner combinations are rejected.
  - Permanent field-owner boundary: field-owned conditions remain same-document-only. They cannot set `sourcePacketTemplateDocumentId` or use a sibling mapping, preserving Step 4a semantics permanently rather than broadening field logic implicitly.
  - Safe sibling-document references: document-owned conditions may reference the owner mapping itself or a sibling `PacketTemplateDocument` only when both mappings belong to the same `PacketTemplate`; references to another packet template, organization, or an arbitrary `DocumentTemplate` are rejected. The `sourceFieldKey` is resolved and operator-checked against the referenced mapping's own `DocumentTemplate`, not the condition owner's template.
  - Inclusion-cycle protection: document-inclusion dependencies are checked deterministically before writes. Direct self-cycles and multi-document cycles are rejected, while acyclic sibling references and non-inclusion purposes remain valid.
  - Evaluator extension: the pure evaluator now accepts `documentFieldValues`, keyed first by `PacketTemplateDocument.id` and then by `fieldKey`, so cross-document leaves resolve without weakening the existing same-document `fieldValues` behavior from Step 4a.
  - Organization-wide dependency protection: field rename/delete checks now find references across every packet template in the organization, not only the field's own document template. The dependency result continues to report accurate counts and purposes. Because this is currently an organization-wide dependency query, query/index scaling should be revisited for organizations with very large template catalogs.
  - `PacketTemplateDocument` dependency helper: a dedicated helper reports conditions that depend on a mapping, supporting safe mapping deletion/repointing decisions without duplicating relationship logic.
  - Standalone packet-template validation: `validatePacketTemplateConditions` validates the entire mapped-document condition graph, including owner/purpose compatibility, sibling membership, referenced-template `fieldKey` resolution, operator/value compatibility, group structure, and inclusion cycles. No packet-template activation/publish lifecycle exists in the current model, so this validation intentionally remains standalone rather than being attached to an invented lifecycle transition; it is not yet called before packet creation.

- Templates & Forms Manager — Stage 5 Step 4c.1: Runtime Identity, Condition Snapshot, and Evaluation Context (identity/snapshot foundation only; no condition behavior is wired into any live path yet)
  - `PacketConditionSnapshot` model: an immutable, definition-only JSON snapshot of a packet's condition graph (mappings, static document/field requiredness, stable field keys and owners, document- and field-owned condition trees with nested `AND`/`OR` groups and cross-document anchors, operators and comparison definitions). It stores a frozen `clientIsMinor` boolean derived once at snapshot time and contains no raw date of birth, no client field values, and no comparison data beyond the condition definitions themselves. No update API exists for a snapshot, so a later template edit cannot retroactively mutate one already created.
  - Legacy vs. condition-aware runtime mode: a packet with both `conditionSnapshotId` and `conditionRuntimeVersion` null is treated as legacy — its composition, validation, completion, editor, and portal behavior stay exactly as they are today, with no snapshot inferred or backfilled and no write performed on an ordinary read. A packet linked to a snapshot is returned in snapshot mode. `createPacket` was not connected to snapshot creation in this step; it only stamps new provenance fields on newly seeded rows.
  - `PacketDocument.packetTemplateDocumentId` and applicability lifecycle fields (`applicabilityStatus`, `conditionallyInactiveAt`, `conditionallyInactiveReason`, all nullable/defaulted) record which mapping a packet document came from and prepare an inactivation state — no inactivation, reactivation, or materialization behavior is implemented yet.
  - `PdfField.templateFieldKey`/`documentTemplateFieldId` provenance: fields seeded from a `DocumentTemplateField` during packet creation now carry their stable source identity forward; manually added fields keep both null, unchanged from today.
  - One-root-per-owner/purpose protection: new database unique constraints on `TemplateConditionGroup` (`documentTemplateFieldId`+`purpose`, `packetTemplateDocumentId`+`purpose`, `validationRuleId`+`purpose`) prevent duplicate root condition groups for the same owner and purpose; nested groups are unaffected since their owner columns are always null. Creation actions reject a duplicate at the application layer first, and now also translate a concurrent-request race that reaches the database constraint (Prisma `P2002`) into the same clean duplicate-root error instead of an unhandled exception. Both existing template validators independently re-detect duplicate persisted roots as defense in depth.
  - Runtime context service (`src/lib/conditions/runtime.ts`, server-only): `createPacketConditionSnapshot` builds and stores the snapshot transactionally after revalidating the packet template's and every mapped document template's condition definitions; `buildPacketConditionContext` resolves a packet into legacy or snapshot mode; `evaluatePacketDocumentInclusion`/`Requiredness`, `evaluatePdfFieldVisibility`/`Requiredness`, and `evaluatePacketApplicability` implement the foundation evaluation semantics (no condition → visible/included by default; no requiredness condition → the static stored value; a hidden field is never effectively required; a manual field is always visible). Raw date of birth is read only transiently while creating a snapshot and is never included in the snapshot, the runtime context, or any audit metadata.
  - Direct runtime indexes: alongside the existing mapping-keyed `packetDocumentsByMappingId` (snapshot mode), the runtime context now also carries `packetDocumentsById` and `pdfFieldsById`, keyed by each row's own database id — the only stable identity a legacy row has, since legacy packets carry no mapping id at all.
  - Corrected legacy requiredness behavior: `evaluatePacketDocumentRequiredness` and `evaluatePdfFieldRequiredness` read a legacy packet's real persisted `PacketDocument.isRequired`/`PdfField.isRequired` directly from the new `packetDocumentsById`/`pdfFieldsById` indexes — never inferred through the mapping-keyed map, which is always empty for legacy packets. This corrects a defect caught during review before this step was approved, where both functions silently fell back to `false` for every legacy packet. Legacy visibility is unaffected (already correctly always `true`, since no condition logic applies to a legacy document/field).
  - Integrity-error behavior: a request for a runtime identity that doesn't exist in the current context (missing packet document, missing PDF field, missing or mismatched snapshot mapping, broken snapshot reference) returns a structured integrity error, never a silent `false`/default. Malformed or inconsistent definitions are surfaced the same way rather than being interpreted as passing or optional.
  - Authorization: every entry point re-verifies staff authentication, organization access, and full parent-chain consistency — packet/client, packet/template, packet/program, mapping/document, and snapshot/template organization matches are all checked before any read or write.
  - Audit: `PACKET_CONDITION_SNAPSHOT_CREATED` records only packet, template, snapshot, and runtime-version identifiers — no DOB, client values, field values, comparison values, or document content.

- Templates & Forms Manager — Stage 5 Step 4c.2a: Transactional Packet Creation, Snapshot Attachment, Initial Document Applicability, and Reconciliation Foundation (`createPacket` is now condition-aware; validation, completion, the PDF editor, and the portal are still untouched)
  - Fully transactional creation: `createPacket` now wraps condition-definition assembly, snapshot creation, the `Packet` row, every `PacketDocument`, every seeded `PdfField`, and all audit events in a single `prisma.$transaction`. A failure anywhere inside — snapshot write, a `PacketDocument` create, or `PdfField` seeding — rolls back the entire attempt; no partial `Packet`, snapshot, document, or field row is ever left behind.
  - Ownership verification before any write: the client, the `PacketTemplate`, the `PacketTemplate`'s program (when set), and every mapped `DocumentTemplate` are each independently re-verified against the caller's active organization — none of it is trusted from the request. Role behavior is intentionally unchanged: `createPacket` remains open to any authenticated org member (not gated by the `canManage()` roles used for template administration), preserving existing case-manager/DSP workflows. A role gate was implemented and then deliberately reverted after review, before this step's commit, to avoid silently narrowing who can create a packet; a dedicated packet-creation permission or configurable role matrix is left as a separate, later design decision.
  - Condition validation gate: `validatePacketTemplateConditions` and `validateTemplateConditions` (for every uniquely mapped document) run before any write; a broken condition definition blocks packet creation entirely with a safe aggregate error count — never silently treated as false, never used to create a partial packet.
  - Not permanently static — the architectural correction: document inclusion/requiredness is evaluated at creation, but a `DOCUMENT_INCLUSION`/`DOCUMENT_REQUIREDNESS` condition that depends on a `TEMPLATE_FIELD` value is classified `unresolved` (every field is still null at creation) rather than guessed. The conservative, compliance-safe default always wins for an unresolved condition: the document is created `ACTIVE` (never hidden for lack of an answer yet), and its static requiredness is preserved unchanged. A condition resolvable from pseudo-fields alone (`$client.is_minor`, `$packet.program_code`, `$packet.packet_type`) is evaluated normally and can genuinely include, exclude, or re-require a document at creation time.
  - Snapshot attachment: the immutable `PacketConditionSnapshot` is created and linked (`Packet.conditionSnapshotId`/`conditionRuntimeVersion`) inside the same transaction, built by a shared `buildPacketConditionDefinition` helper (extracted from Step 4c.1's `createPacketConditionSnapshot`, which is otherwise unchanged and still independently tested). Every new packet becomes condition-aware from this point forward, even when its template has no conditions at all — only packets created before this step remain legacy.
  - Reconciliation foundation without a schema change: no new column stores "pending reconciliation" state, because it is always re-derivable from the snapshot definition plus current field values — the same reason field-level visibility never needed storage in Step 4c.1. `evaluateInitialPacketApplicability` (creation-time, pure, no Prisma) and `determinePacketReconciliationNeeds` (general-purpose, works against any live snapshot-mode runtime context at any later time) share one classification primitive and form the internal API boundary Stage 5 Step 4c.2b will build actual re-evaluation on top of. Neither performs automatic re-evaluation, writes on page load, or hooks into field saves — that remains 4c.2b's job.
  - Stable creation-time context: `clientIsMinor` is frozen via the same derivation as Step 4c.1, evaluated at one fixed reference timestamp shared by both the snapshot and the packet's own `createdAt`; `Packet.programId` is now set from `PacketTemplate.programId`; program code is read from the stable `Program.code`, never display text.
  - Provenance unchanged in shape: every created `PacketDocument` stores `packetTemplateDocumentId`, `applicabilityStatus` (always `ACTIVE` from this function — `CONDITIONALLY_INACTIVE` stays reserved for a future reconciliation pass), and static `isRequired`/`sortOrder`; every seeded `PdfField` carries `templateFieldKey`/`documentTemplateFieldId`/`source: "template"`/`value: null` exactly as Step 4c.1 defined.

- Templates & Forms Manager — Stage 5 Step 4c.2b: Live Document Applicability Reconciliation on Field Save (`saveDocumentFields` now closes the reconciliation loop for document inclusion; requiredness integration, validation, completion, editor UI, and the portal remain untouched)
  - Not permanently static, closed: cross-document conditions exist specifically so a document's applicability can change once a controlling field is actually answered. Every successful `saveDocumentFields` call on a condition-aware packet now re-evaluates `DOCUMENT_INCLUSION` for every mapped document against the packet's frozen snapshot and the just-written field values, and persists any real transition to `PacketDocument.applicabilityStatus`.
  - Fully transactional save: field-ownership verification, every field upsert, the edited document's status recalculation, condition reconciliation, and every audit event now commit inside one `prisma.$transaction` — a failure anywhere (including a reconciliation integrity error) rolls back the whole save, leaving prior field values and document states untouched. Previously, field writes happened one at a time outside any transaction.
  - **Fixed a pre-existing security gap while making this transactional**: `saveDocumentFields` previously updated `PdfField.value` by client-supplied id with no ownership check at all — any authenticated editor could have overwritten a field belonging to a different document, packet, or organization. Every submitted field id is now verified against the target document's own fields before any write; an unrecognized id rejects the entire save with a generic error, never revealing which ids are valid.
  - `reconcilePacketDocumentApplicability` reads exclusively through the same transaction client that just wrote the new field values (never the global Prisma client, which would not see this transaction's own uncommitted writes) and evaluates only against the packet's immutable `PacketConditionSnapshot` — later edits to live template conditions can never affect an already-created packet's reconciliation.
  - Bidirectional, idempotent state machine: inclusion evaluating true reactivates a `CONDITIONALLY_INACTIVE` document (clearing `conditionallyInactiveAt`/`conditionallyInactiveReason` to `null`); inclusion evaluating false deactivates an `ACTIVE` document (`conditionallyInactiveAt: now()`, `conditionallyInactiveReason: "condition_false_after_field_save"`, a safe machine-readable value only); a state that already matches the desired outcome triggers no write and no audit event. Only `PacketDocument.applicabilityStatus` and its two inactivity fields are ever touched — `PdfField` rows, `PacketDocument.isRequired`, and `PdfField.isRequired` are never modified by reconciliation.
  - Reconciliation is deliberately narrow in scope: it only ever transitions an **already-materialized** `PacketDocument` between `ACTIVE` and `CONDITIONALLY_INACTIVE`. It never creates a new `PacketDocument` — if a mapping evaluates applicable but has no corresponding row (which Step 4c.2a's conservative creation-time policy should make impossible), that is treated as an integrity error that aborts the entire field save rather than silently materializing a document mid-save.
  - Legacy packets (no condition snapshot) are completely unaffected: `saveDocumentFields` checks `conditionSnapshotId`/`conditionRuntimeVersion` once and skips reconciliation entirely for them — fields save and the document's static-requiredness-based status recalculates exactly as it always has.
  - Requiredness is explicitly out of scope for this step: the edited document's status recalculation still counts against static `PdfField.isRequired` only, unchanged from before — effective, condition-aware requiredness remains a future integration, not something this step claims.
  - Audit: new `PACKET_DOCUMENT_APPLICABILITY_RECONCILED`, one event per real transition only, metadata limited to packet/document/mapping ids, previous/next `applicabilityStatus`, and `trigger: "field_save"` — no field values, comparison values, or client data.
  - Verification: `npx prisma validate`, `npm run lint`, `npx tsc --noEmit`, and `npm run build` all clean; `npm test` **545/545 passed**. Live-verified against real Postgres and the real running server: built a real two-document packet where the second document's inclusion depends on a checkbox on the first; checking the box kept it active, unchecking it flipped the real row to `CONDITIONALLY_INACTIVE` with a real timestamp and reason, re-checking it reactivated the same row with the timestamp/reason cleared, and a third, unchanged save produced zero new database rows and zero new audit events — confirming idempotency against real data, not just mocks. A fabricated foreign field id was correctly rejected by the new ownership check.
  - Deferred to a later step: effective/condition-aware requiredness integration, validation-engine integration, completion/readiness integration, the PDF editor's field visibility and conditional-requiredness UI, and portal applicability filtering.
